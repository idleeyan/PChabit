## 第三次发布（pchabit.ico 修复）

**问题**: 用户报告"程序运行正常，只有 ico 图标没有改变，但其他图标已经改变"

**根本原因**:
1. csproj 中 `<Content Include="pchabit.ico"><CopyToOutputDirectory>PreserveNewest</CopyToOutputDirectory></Content>`
2. `PreserveNewest` 行为：源新于目标才覆盖。当 src/pchabit.ico 和 pub/pchabit.ico mtime 相同时，Publish 不会复制
3. 用户看到的"其他图标"（如 Square44x44Logo.png）变了，是因为它们是 PNG，运行时通过 XamlRoot 加载，WinUI 不会缓存
4. `pchabit.ico` 是 Windows 资源级别（csproj 行 18 `<ApplicationIcon>`），嵌入到 PChabit.exe 的资源段，Windows 在 IconCache.db 中缓存

**修复**:
1. 删除 src/PChabit.App/pchabit.ico
2. 重新复制 scripts/icons-tray/tray-running.ico → src/PChabit.App/pchabit.ico (mtime 刷新到 11:48)
3. 重新 `dotnet publish` → publish/pchabit.ico mtime 11:48:19，内容是新图标
4. MD5 哈希一致：df49f833f77d197833b7de259ea9a6b2

**附加问题**: Windows IconCache.db 可能仍缓存旧图标，用户需要：
  - 重启 Explorer
  - 或运行 `ie4uinit.exe -show` 刷新缓存
  - 或重启系统

**项目规则补充（关键）**:

## 数据管理页面打开卡死修复 (v2.26.1)

**用户报告**: 打开"数据管理"页面卡顿至死

**根本原因**（双重卡死）:
1. `OnNavigatedTo` 中 `await ViewModel.LoadBackupsAsync()` 同步等待
2. `LoadOverviewAsync` 内部 `await` 5 个 EF Core 查询（4 个 Count + Min + Max）
3. **关键**: WinUI 3 一次性绑定 5 个 Border + 6 个 TextBlock + 2 个 ListView + 12 个其他 UI 元素，UI 线程在 x:Bind 初始化时被同步等待

**XAML 编译顺序的微妙陷阱**:
- x:Bind 在 Page 构造时立即建立绑定，TextBlock 一旦绑定到 ViewModel 属性就触发 PropertyChanged 订阅
- 第一次 PropertyChanged 在 LoadOverviewAsync 执行后立即触发，会引发 12 个 UI 元素的 InvalidateArrange
- DB 查询慢 + UI 重新布局慢 = 双重死锁

**修复方案**（彻底解决）:
```xml
<Grid>
    <ScrollViewer x:Name="ContentScrollViewer" Visibility="Collapsed">
        ... 全部内容 ...
    </ScrollViewer>
    <ProgressRing x:Name="LoadingRing" IsActive="True" />
</Grid>
```
```csharp
public DataManagementPage()
{
    InitializeComponent();
    ...
    Loaded += DataManagementPage_Loaded;  // 不用 OnNavigatedTo
}

private async void DataManagementPage_Loaded(object sender, RoutedEventArgs e)
{
    LoadingRing.IsActive = true;
    await Task.Run(async () =>
    {
        await ViewModel.LoadOverviewAsync();
        await ViewModel.LoadBackupsAsync();
    });
    LoadingRing.IsActive = false;
    ContentScrollViewer.Visibility = Visibility.Visible;
}
```

**关键决策**:
- ContentScrollViewer 初始 `Collapsed` — 完全不渲染内部 DOM
- Loaded 而非 OnNavigatedTo — Loaded 触发时 Page 已 Attached to Visual Tree，ProgressRing 能正常显示
- `Task.Run` 把 DB 查询丢到线程池，UI 线程 0 阻塞
- 一次性显示 ContentScrollViewer，避免多次 InvalidateArrange

**预防规则**:
- 任何包含 DB 查询 + 多个 ListView 的页面，必须用「ProgressRing 占位 + Loaded 后 Task.Run + 一次性显示」
- x:Bind 大量属性绑定的页面尤其需要这个保护
- OnNavigatedTo 中**禁止** `await` 耗时 DB 查询

**已应用此模式**: DataManagementPage v2.26.1
**未来需要同样改造**: 任何 Page 包含 >3 个 ListView 或 >2 个 DB 表查询

## 数据管理页面卡死 v2.26.1 修复无效的根因 (v2.26.2)

**用户反馈**: "问题依旧" — v2.26.1 的 ProgressRing 占位 + Loaded 事件方案没有解决问题

**真正的根因**（最隐蔽的反模式）:
WinUI 3 默认带 `SynchronizationContext`。`await ViewModel.LoadDataAsync()` 看似异步，但 EF Core 内部 `Task.GetAwaiter().GetResult()` 会在 UI 线程上同步等待 DB 查询完成。**ProgressRing 占位 + Loaded 事件只是把"卡死"从"无 UI"变成"有 UI"，但 DB 查询本身仍在 UI 线程同步等待。**

**v2.26.2 真正的修复**:

1. **ViewModelBase 新增 `LoadInBackgroundAsync` 统一包装**:
   ```csharp
   public async Task LoadInBackgroundAsync(Func<Task> loadAction)
   {
       await Task.Run(async () => await loadAction());  // 强制进线程池
   }
   ```
   `Task.Run` 会确保 `loadAction` 在线程池执行，EF Core 的内部 `GetResult()` 也会在线程池上同步等，不再阻塞 UI 线程。

2. **DataManagementViewModel.LoadOverviewAsync 合并 6 查询为 1 SQL**:
   ```csharp
   var stats = await dbContext.Database.SqlQueryRaw<DbStats>(@"
       SELECT 
           (SELECT COUNT(*) FROM AppSessions) AS AppCount,
           (SELECT COUNT(*) FROM KeyboardSessions) AS KeyCount,
           (SELECT COUNT(*) FROM MouseSessions) AS MouseCount,
           (SELECT COUNT(*) FROM WebSessions) AS WebCount,
           (SELECT MIN(StartTime) FROM AppSessions) AS Earliest,
           (SELECT MAX(StartTime) FROM AppSessions) AS Latest
   ").FirstAsync();
   ```
   6 次往返 → 1 次往返，SQLite 锁冲突概率降低到 1/6。

3. **系统性地应用 LoadInBackgroundAsync 到所有页面**:
   Dashboard / Timeline / KeyboardDetails / Insights / Analytics / Heatmap / DataManagement 全部改用 `LoadInBackgroundAsync` 模式。

**WinUI 3 异步陷阱（必须记住）**:
- ❌ 任何 `await ViewModel.LoadXxxAsync()` 直接写在 `OnNavigatedTo` 都会卡死
- ❌ `await Task.Run(...)` 在 `Loaded` 事件中如果 loadAction 内部又 await 会回到 UI 线程
- ✅ 必须 `await ViewModel.LoadInBackgroundAsync(async () => await ViewModel.LoadXxxAsync())` 双重包裹
- ✅ DB 查询语句合并为单次 `SqlQueryRaw` 减少往返次数
- ✅ ViewModel 内部用 `IDbContextFactory` + 独立的 DbContext 实例（不要共享）
