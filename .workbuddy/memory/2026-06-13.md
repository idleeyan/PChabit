### P0 关键优化
1. **DataCollectionService 批量处理** — `ProcessDataAsync` 从每次操作创建独立 Scope 改为批量50个操作共享一个 Scope + 单次 SaveChanges；`SaveBackgroundSessionsPeriodicallyAsync` 从 N 次独立 Scope 改为单个 Scope 批量提交；`FlushAccumulatorsAsync` 保留单 Scope 但将累加器快照与数据库操作分离

### 重要修复2：SQLite 连接字符串不支持连接池
- **问题**: P1-8 优化中添加了 `Pooling=true;Max Pool Size=10;`，但 SQLite (Microsoft.Data.Sqlite) 不支持这些关键字
- **影响**: 所有数据库操作全部失败，所有监控数据无法写入，所有页面查询报错
- **修复**: 移除 `Pooling=true;Max Pool Size=10;`，SQLite 使用 `Cache=Shared` 即可实现共享连接
- **教训**: SQLite 不支持 SQL Server 的连接池关键字，只能用 `Cache=Shared` 实现多连接共享

### 键鼠统计查询修复 - 真正的根因 (v2.22.3, 2026-06-13 18:00)

v2.22.2 修复了钩子异常保护和健康检查，但键鼠统计依旧失效。

#### 真正的根因：EF Core DateTime 格式不匹配

排查发现数据库中 Date 存储为 `2026-06-13 00:00:00`（19字符，无时区后缀），
但 EF Core 对 DateTimeKind.Local 的 DateTime 参数会附加 `+08:00` 时区后缀（如 `2026-06-13 00:00:00+08:00`）。

SQLite 执行 TEXT 字符串比较时：
- `'2026-06-13 00:00:00' == '2026-06-13 00:00:00+08:00'` → FALSE
- `'2026-06-13 00:00:00' >= '2026-06-13 00:00:00+08:00'` → FALSE（短字符串 < 长字符串）

AppSessions 的范围查询碰巧能工作，因为 StartTime 有非零小时部分（如 `17:37:58`），字符串比较在小时位就胜出了。
但 KeyboardSessions/MouseSessions 的 Date 全是午夜 `00:00:00`，比较在时区后缀处失败。

验证测试：
- `Date >= '2026-06-13 00:00:00+08:00'` → 0 条（FAIL）
- `Date >= '2026-06-13 00:00:00'` → 3 条（OK）

#### 修复内容

1. **DbContext 值转换器**: 为 KeyboardSession/MouseSession/DailyPattern/EfficiencyScore/WorkPattern 的 Date 属性添加
   `v => v.ToString("yyyy-MM-dd HH:mm:ss")` / `v => DateTime.SpecifyKind(DateTime.Parse(v), DateTimeKind.Unspecified)`
   确保写入和查询参数格式一致（无时区后缀），无需数据迁移

2. **15+ 文件查询修复**: 所有 `s.Date == date` 和 `s.StartTime.Date == date.Date` 改为范围查询
   - `s.Date == today` → `s.Date >= today && s.Date < tomorrow`
   - `s.StartTime.Date == date.Date` → `s.StartTime >= date.Date && s.StartTime < date.Date.AddDays(1)`
   - EF Core SQLite 不支持 DateTime.Date 属性翻译，范围查询更健壮
   - 涉及文件：DashboardViewModel, KeyboardDetailsViewModel, MouseDetailsViewModel, DetailDialogViewModel,
     KeyboardSessionRepository, MouseSessionRepository, DailyPatternRepository, PatternAnalyzer,
     InsightService, EfficiencyCalculator, GoalService, BehaviorAnalyzer, SessionAggregator,
     PatternDetector, DailyAggregator

### 键鼠统计第三次修复 (v2.22.4, 2026-06-13 18:20)

v2.22.3 修复了 EF Core DateTime 格式问题，但用户反馈"问题依旧"。

#### 新发现的根因

1. **EF Core 并发查询**: DashboardViewModel 和 KeyboardDetailsViewModel 使用 Task.WhenAll 在同一个 DbContext 上并发查询
   - EF Core 不支持同一 DbContext 实例上的并发操作，会抛 InvalidOperationException
   - 异常被 catch 吞掉，键鼠数据全部为空

### 键鼠统计终极修复 (v2.23.0, 2026-06-13 19:25)

v2.22.2~v2.22.5 四次修复均未解决问题。通过与 git 中 v2.15 原始代码对比，发现**真正的根因**：

#### 根因分析

v2.15 原始 DataCollectionService 使用 `EnqueueOperation` 逐事件写入数据库（每次键盘/鼠标事件直接入 Channel，消费者读出后立即写 DB）。
后续性能优化将其改为 accumulator+FlushTimer 模式（5秒间隔批量 flush），但这个改造存在致命缺陷：
- flush 可能因异常/竞态静默丢失所有累积数据
- flush 周期内应用崩溃则全部数据丢失
- accumulator 的 GetByDateAndHourAsync 在没有 DateTime 值转换器时始终返回 null（导致每5秒创建新 session 而非更新现有 session）

#### 修复内容

1. **DataCollectionService 完全回退到 EnqueueOperation 模式**:
   - 删除 KeyboardSessionAccumulator / MouseSessionAccumulator / FlushAccumulatorsAsync / _flushTimer
   - OnKeyboardDataCollected / OnMouseClick / OnMouseMove / OnMouseScroll 改回 v2.15 的 EnqueueOperation 逐事件写入
   - ProcessDataAsync 消费者批量50操作/Scope，但每个操作独立 try-catch，一个失败不影响其他
   - 保留 UpdateTypingSpeed 方法

2. **诊断日志降级**: KB-Start/MS-Start/KB-Event/MS-Event 从 Information 降为 Debug，避免生产环境日志膨胀
